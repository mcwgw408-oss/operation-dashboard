# design-decisions.md — 設計判断の理由集

決定そのものはコードとdocsに残る。この文書が残すのは**理由**と、**却下した案**。
将来の設計者が同じ検討をやり直したり、知らずに逆走したりしないための記録。

---

## 画面・生活導線

**D1. 4ゾーンの縦一列＋折りたたみ。タブ分離は却下。**
理由: タブは朝の対話を分断するリスクと、切り替え操作という日々のコストを生む。縦一列なら「スクロール＝対話の進行」になる。却下したタブ案が有利になるのは画面数が大きく増えた場合のみ。

**D2. 並び順はCSSのorderプロパティで一元管理。DOM移動は最小限。**
理由: app.jsがID参照でDOMを掴むため、DOM移動は参照破壊のリスクを伴う。order値ならCSS1行で並び替えられ、保留事項（位置未確定のパネル）を低コストで持ち越せる。ゾーンごとに番号帯を分ける（ゾーン1=10番台、2=30番台、3=60番台、4=80番台）。
注意: デフォルトorderの受け皿（`.dashboard > * { order: 20 }`）は全要素の明示指定が完了した時点で削除済み。**新規要素には必ずorderを指定する。**

**D3. 朝の対話の一直線は最重要不変条件。**
「ひとこと→集中→（意図）→提案→候補にした理由→返答ボタン→生成された返答→自然/違和感→メモ」の間に、他の要素を挟まない。押した直後、視線のすぐ下に応答が現れる報酬構造を守る。
関連する罠: `moveDashboardNode` による実行時移設のため、**DOMの並び＝画面の並びではない**。判定は必ず実行時の並びで行う。

**D4. 折りたたみの基準は「毎日入力するか」であって「重要か」ではない。**
実例: 学びの蓄積は最重要だが折りたたまない場所へ、発信アイデアは地味だが毎日書くので展開のまま。この基準変更が生活導線設計の核心だった。

**D5. 感情設計は余白・強弱・言葉で作る。装飾は足さない。**
- 区切りは箱ではなく「素の文字＋大きな余白」。カードの海では箱は区切りにならない
- 朝いちばんに進捗0%を見せない（ひとことが最初の声）
- 下へ行くほど静かに。ゾーン見出しは呼びかけの言葉（「今日のはじまり」等、確定済み）
- 5年耐性: 本文14px以上・muted色はコントラスト確保・枠と影の二重装飾は避ける
- 「飽きない」の正体は器の透明化。中身（さくらの提案と日々の言葉）が毎日変わるから、器は変わらなくていい。**飽き対策の装飾追加は却下**

## データ

**D6. 保存形式は変更禁止・追加のみ。**
既存データが読めなくなる変更は理由を問わず不可。スキーマバージョンは後方互換の追加では上げない（上げると逆に互換を壊す）。

**D7. 新しい保存キーは、同じコミットでBACKUP_KEYSへ追加。**
起源: かつて学習ログ（sakura-learning-log-v1）と記憶ストア（sakura-memory-store-v1）がバックアップ対象から漏れており、端末変更で「さくらが記憶喪失になる」状態だった。修正後、静的検査が網羅性を自動監視している。**守りたいルールは検査に翻訳する**——これがこのプロジェクトの保守の基本姿勢。

**D8. 復元は「ファイルにあるキーだけ上書き」（マージ的復元）。**
古いバックアップで復元しても、ファイルに無いキーの手元データは消えない。完全初期化してからの復元は別の仕組みが必要（未実装・必要が生まれたら設計）。

**D9. 6つのLaboは同一オリジンでlocalStorage容量（目安5〜10MB）を共有している。**
GitHub Pagesの構造上の事実。容量の見積もりはダッシュボード単体で考えない。追記型ログの上限方針は未決（open-questions.md）。

## コード構造

**D10. file直開き（file://）運用を守る。したがってES Modules分割は不可。**
分割は「従来型scriptタグを依存順に並べ、グローバル共有のまま5〜8ファイル」方式のみ。ビルド環境の導入・フレームワーク化は却下（静的HTML+JSがこの規模と運用の正解）。

**D11. 大規模リファクタは、地図とテストが揃ってから、単独Epicで。**
順序: app-js-map（区画の文書化）→ゴールデンテスト→機械的分割。機能変更と絶対に混ぜない。「変える前に、変わっていないことを証明する道具を作る」。

## 判断パイプライン（さくらの頭の中）

**D12. 直列パイプラインは却下。4段の漏斗を採用。**
検討された「健康→予定→記憶→学び→発信→学習→提案」の直列案は、独立した証拠を変換の連鎖として並べるため、恣意的な順序決定が7箇所生まれる。採用した構造: **収集（並列に読み1つのコンテキストへ）→解釈（信号の導出）→決定（候補生成・採点・選択）→表現（言葉にする）**。実装: collectBrainContext → buildBrainDecision → buildBrainExpression。

**D13. 健康は2回登場するのが正しい。**
決定段では「制約」（器の大きさ）、表現段では「声の温度」。先頭で1回だけ通す案は2つの役割を混ぜるため却下。

**D14. 提案学習は「段」ではなく「重みと外側のループ」。**
朝の判断では採点係数として読むだけ。書き込みはフィードバック後（パイプラインの外）。1回の判断の中でデータは一方向に流れる。

**D15. 表現の分離は「さくらの声の編集室」を作る工事。**
判断と説明が別コードで組み立てられていると、実際の判断と説明文がズレる。パイプラインの各段の入出力記録がそのまま「なぜこの提案」になる構造を目指す。

## 意図（Cockpit）

**D16. 憲法: 「意図は命令ではなく、今日の方向を示すもの。AIは方向を尊重し、実行可能な歩幅だけを提案する。」**（さくら確定・2026-07-09）

- 意図は**朝、その日の自分を確認したうえで**書く（前夜の計画ではない）。その朝の意図は状態を織り込み済みなので、AIが体調でさらに縮めると二重調整になる
- **空欄が正常。** 催促・空欄への言及は一切しない。書かない朝は体調と状況が舵を取る（既存パイプライン）
- 意図がある朝: 方向は意図のまま。体調は「安全網」（見立てと記録が大きく食い違う朝の確認）と「歩幅の調整」のみ
- **拒否権としての意図は却下**（書いたことをそのまま返す鏡はAIの意味がない）。**意図による他材料の全無視も却下**
- 方向の復唱は原文のまま（要約・言い換えをしない）
- 安全確認の確定文言（一字一句変更禁止）: 「その方向でいきましょう。今日は少し小さめの入口から始めると良さそうです。」
- 透明性: 意図どおりにしなかった調整を黙らない。人が意図を無視されたと感じるのは、変えられたときではなく黙って変えられたとき
- 再生成の境界: 意図保存→提案を作り直す。最初の返答ボタン以降は作り直さない
- **決めすぎない**: 履歴・一覧・振り返り・保持期間・Snapshot連携は、必要が生まれた段階で設計する（「載せない」と決めたのでもない）

## 体制

**D17. データに関する判断（間引き・変換・移行）はCodexに委ねない。**
方針の言語化は設計役、承認はさくら、実装はCodex。この順序を崩さない。

**D18. 判断の重み・縮尺・声の温度は技術判断ではない。**
「さくらAIにどう在ってほしいか」の判断であり、最終決定者は常にさくら。設計役は選択肢と影響を言語化する役。
