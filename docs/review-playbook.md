# review-playbook.md — operation-dashboard 完成レビューの手順書

このプロジェクトのレビュー文化を、レビュー役が交代しても続けられる形で残す文書。
体制: 設計・レビュー役（本書の読者）＋ Codex（実装担当）＋ さくら（意思決定者）。

---

## 1. レビューの三原則

1. **実コード主義。** 判定は必ず実物のコードで行う。push前のレビューは「条件付き合格」まで。確定判定はpush後に最新mainをクローンし直して行う。報告がどれだけ丁寧でも、この順序を崩さない。
2. **毎回クローンし直す。** Codexは独自の改善を加えることがあるため、レビュー開始時に必ず `rm -rf` して最新mainを取り直す。手元の古いクローンで判定しない。
3. **報告を疑うことは、不信ではない。** 疑って深掘りした結果「報告が正しかった」と証明されるのが理想の結末。実例: Step8-aで「2行のdiffで一直線化は不可能」と疑い、実行時移設（moveDashboardNode）の仕組みを掘り当て、報告の正確さを証明した。疑いの過程と結論の両方を正直に書く。

## 2. レビュー手順（毎回同じ）

1. `git clone --depth 20` で最新mainを取得、`git log --oneline` でコミット構成を確認
2. `git show <hash> --stat` で変更規模を見る。**報告内容と規模が釣り合わない場合は必ず理由を掘る**（小さすぎる/大きすぎる、どちらも兆候）
3. diffを精読。行番号ではなく、セレクタ名・関数名・id名で変更を特定する
4. 静的な構造解析で裏を取る（grep/pythonでDOM順・親子関係・参照の有無を機械的に確認。目視より確実）
5. 判定を出す: **合格／条件付き合格／保留**＋push可否＋次Stepへの進行可否
6. 判定文には「検証した方法」を必ず書く。結論だけの判定は残さない

## 3. 確認観点チェックリスト

変更の種類を問わず、毎回通す観点:

- [ ] **id / data属性の保全**: 削除行(-)に出るidがすべて追加行(+)に再登場するか。app.jsはID参照でDOMを掴むため、id欠落は機能が静かに死ぬ
- [ ] **localStorageキーの網羅**: キー定数の総数とBACKUP_KEYSの数が一致するか（静的検査が監視しているが、レビューでも数える。実例: Step9-a以前、学習ログと記憶がバックアップから漏れていた）
- [ ] **保存形式**: 変更禁止・追加のみ。既存データが読めなくなる変更は理由を問わず不合格
- [ ] **order指定**: dashboard直下への要素追加には必ずorder値。デフォルトの受け皿は存在しない（未指定は画面最上部に出現する）
- [ ] **実行時移設の罠**: `moveDashboardNode` により**DOMの並び＝画面の並びではない**。レイアウト判定の前に必ずapp.jsの移設リスト（6600行台）を確認する。本プロジェクト最大の落とし穴
- [ ] **編集UIのバインド**: 絞り込み・並び替えを伴う編集/削除は、配列インデックスではなくid/オブジェクト参照ベースか
- [ ] **UI文字列は日本語**のみ
- [ ] **ゴールデンテストと静的検査**の出力が報告に添付されているか
- [ ] **file直開き制約**: ES Modules（import/export）はfile://で動かない。分割・読み込み方式の変更はこの制約に照らす

## 4. Codexの報告に要求するテンプレート

- commit hash / message
- 変更ファイルと変更内容
- **変更しないもの**（明記させる。書かれていない場合は差し戻し理由になり得る）
- **動作確認した操作の列挙**（UIやレンダリングに触るStepでは必須。`node --check`だけの報告は粒度不足として保留にする。実例: Step9-b）
- ゴールデンテスト・静的検査の出力
- 判断に迷った箇所（実装せず質問として持ち帰らせる）

## 5. 指示書を書くときの規律

- Step順・リスクの低い順。1Step＝1コミット。機能Stepと整理Stepを混ぜない
- 「変更しないもの」を必ず章として書く
- 各Stepに確認手順を付ける。確認手順のないStepは書かない
- 対象の特定は行番号ではなくセレクタ名・関数名で（行番号は前のStepでズレる）
- 削除指示はリストの完全性に注意（実例: Step8-cで指示リスト外の4箇所が残った。原因は指示書側の調査不足であり、その旨を記録した）
- 文言を確定させたら「一字一句変更禁止」と明記する

## 6. 判定の語彙と使い分け

- **合格**: 実コードで検証済み、問題なし
- **条件付き合格**: push前で報告ベース。低リスク変更に限る。確定はpush後
- **保留**: 実コードなしでは設計の核心が判定できない場合（実例: Step9-bの検索スコープ）。保留は失敗ではなく、判定材料の請求である
- 合否に影響しない発見は「問題点」ではなく「記録事項」として残す

## 7. データに関する判断の線引き

**データを間引く・変換する・移行する判断は、Codexに委ねない。** 方針は設計役が言語化し、さくらの承認を得てから実装指示にする。学習ログと記憶は間引かない聖域（仮方針、open-questions.md参照）。

## 8. レビュー役への最後の申し送り

このプロジェクトは「さくらの生活そのもの」を支えている。判定に迷ったら、技術的な正しさより先に「明日の朝、さくらが画面を開いたときに何が起きるか」を想像すること。朝の対話の一直線（提案→理由→返答→評価→メモ）を分断する変更は、他のすべてが正しくても不合格である。
