# Sakura Snapshot JSON Specification

Version: Phase1 / snapshotVersion 1

This document defines the JSON generated by the operation-dashboard Sakura AI panel. The snapshot is a read-only handoff format for AI use. It must not be treated as a backup or import format.

## Top-Level Object

```json
{
  "format": "sakura-snapshot",
  "snapshotVersion": 1,
  "dictionaryVersion": "v1.3",
  "createdAt": "2026-07-03T07:30:00.000Z",
  "mode": "morning",
  "period": {
    "detailDays": 7,
    "from": "2026-06-27",
    "to": "2026-07-03"
  },
  "privacy": {
    "reflection": true,
    "feelings": true,
    "mailDmCounts": true,
    "stock": false
  },
  "summary": {
    "todayProgress": "3/16",
    "seedsFermenting": 4,
    "longestFermentingDays": 21,
    "revisitPeople": 3,
    "writingInProgress": 2,
    "openNextActions": 5
  },
  "apps": {
    "operation-dashboard": {
      "schemaVersion": 1,
      "data": {}
    },
    "discovery-labo": {
      "schemaVersion": 1,
      "data": {}
    },
    "koryu-log-labo": {
      "schemaVersion": 1,
      "data": {}
    },
    "hasshin-kansatsu-labo": {
      "schemaVersion": 1,
      "data": {}
    },
    "substack-labo": {
      "schemaVersion": 1,
      "data": {}
    },
    "stock-labo": null,
    "operation-cockpit": {
      "schemaVersion": 1,
      "data": {
        "recentDays": {}
      }
    }
  }
}
```

## Top-Level Keys

| Key | Type | Required | Description |
|---|---|---:|---|
| `format` | string | Yes | Fixed value: `sakura-snapshot`. Identifies this file as an AI handoff snapshot. |
| `snapshotVersion` | number | Yes | Snapshot envelope version. Current value: `1`. |
| `dictionaryVersion` | string | Yes | Data dictionary version used to interpret the snapshot. Current value: `v1.3`. |
| `createdAt` | string | Yes | ISO 8601 timestamp generated when the snapshot is created. |
| `mode` | string | Yes | Snapshot mode. Allowed values: `morning`, `night`. |
| `period` | object | Yes | Date range used for detailed dashboard data. |
| `privacy` | object | Yes | Privacy switches applied at generation time. |
| `summary` | object | Yes | Computed summary values for quick AI reading. |
| `apps` | object | Yes | Per-Labo payloads. Each app value is either an app envelope or `null`. |

Unknown future top-level keys must be ignored by readers unless their meaning is explicitly documented in a later `snapshotVersion`.

## `period`

| Key | Type | Required | Description |
|---|---|---:|---|
| `detailDays` | number | Yes | Number of recent days included in detailed operation-dashboard data. Current value: `7`. |
| `from` | string | Yes | Start date in `YYYY-MM-DD` format. |
| `to` | string | Yes | End date in `YYYY-MM-DD` format. Usually the local date at generation time. |

`period` applies directly to `apps["operation-dashboard"].data.recentDays`. Other apps can use different selection rules.

## `privacy`

| Key | Type | Required | Default | Description |
|---|---|---:|---:|---|
| `reflection` | boolean | Yes | `true` | Whether operation-dashboard reflection text is included. |
| `feelings` | boolean | Yes | `true` | Whether selected feeling-related fields from koryu-log-labo are included. |
| `mailDmCounts` | boolean | Yes | `true` | Whether selected mail and DM numeric fields are included. |
| `stock` | boolean | Yes | `false` | Whether stock-labo data is included. |

These values are copied from `sakura-snapshot-settings-v1` at generation time. Readers must treat `false` as intentional omission, not as missing or empty data.

## `summary`

| Key | Type | Required | Nullable | Description |
|---|---|---:|---:|---|
| `todayProgress` | string | Yes | No | Completed/tracked count for today's dashboard items, formatted as `done/total`. |
| `seedsFermenting` | number | Yes | No | Count of discovery records whose `status` is `発酵中`. |
| `longestFermentingDays` | number or null | Yes | Yes | Maximum number of days since `statusChangedAt` among fermenting discoveries. `null` when no computable fermenting record exists. |
| `revisitPeople` | number | Yes | No | Count of unique koryu-log names with `revisit` equal to `はい`. |
| `writingInProgress` | number | Yes | No | Count of Substack writing items with `status` equal to `執筆中`. |
| `openNextActions` | number | Yes | No | Count of hasshin-kansatsu records with a non-empty `nextAction`. |

Summary values are generated from localStorage data at snapshot creation time. They are convenience values, not authoritative source data.

## `apps`

The `apps` object has one key per Labo. Each app value is either:

```json
{ "schemaVersion": 1, "data": {} }
```

or `null`.

`schemaVersion` describes the app payload shape, not the snapshot envelope. For Phase1, all non-null app envelopes use `schemaVersion: 1`.

### `apps["operation-dashboard"]`

Required: Yes

Nullable: No

```json
{
  "schemaVersion": 1,
  "data": {
    "recentDays": {},
    "olderDaysCount": 0,
    "laterItems": [],
    "persistentMemos": []
  }
}
```

| Key | Type | Required | Description |
|---|---|---:|---|
| `recentDays` | object | Yes | Map of `YYYY-MM-DD` to daily dashboard records for the last `period.detailDays` days. |
| `olderDaysCount` | number | Yes | Count of dashboard day records older than `period.from`. Older records are not included in detail. |
| `laterItems` | array | Yes | Incomplete items from `operation-dashboard-later-v1`. Items with `done: true` are excluded. |
| `persistentMemos` | array | Yes | All persistent memo records from `operation-dashboard-persistent-memos-v1`. |

Daily records inside `recentDays` follow the existing `operation-dashboard-v1` shape. The snapshot does not normalize or migrate them.

Null conditions inside daily records:

| Field | Null Condition |
|---|---|
| `reflection` | Set to `null` when `privacy.reflection` is `false`. |
| `metrics.dmPending` | Set to `null` when `privacy.mailDmCounts` is `false`. |
| `metrics.dmHandled` | Set to `null` when `privacy.mailDmCounts` is `false`. |
| `metrics.mailUnread` | Set to `null` when `privacy.mailDmCounts` is `false` and the field exists. |
| `metrics.mailProcessed` | Set to `null` when `privacy.mailDmCounts` is `false` and the field exists. |

Mail check booleans such as `mailMorningChecked`, `mailNoonChecked`, `mailNightChecked`, and `dmPreviousDone` currently remain present even when `privacy.mailDmCounts` is `false`.

### `apps["discovery-labo"]`

Required: Yes

Nullable: No

```json
{
  "schemaVersion": 1,
  "data": {
    "discoveries": [],
    "sources": []
  }
}
```

| Key | Type | Required | Description |
|---|---|---:|---|
| `discoveries` | array | Yes | All records from `discovery-labo-discoveries`. |
| `sources` | array | Yes | All records from `discovery-labo-entry-sources-v1`. |

No privacy switch currently nulls discovery-labo fields.

### `apps["koryu-log-labo"]`

Required: Yes

Nullable: No

```json
{
  "schemaVersion": 1,
  "data": {
    "entries": []
  }
}
```

Selection rule:

- Include entries whose `date` is within the last 30 days.
- Also include entries outside that range when `revisit` is `はい`.

Null conditions:

| Field | Null Condition |
|---|---|
| `tension` | Set to `null` when `privacy.feelings` is `false`. |
| `impression` | Set to `null` when `privacy.feelings` is `false`. |
| `happyMoment` | Set to `null` when `privacy.feelings` is `false`. |

Other fields are copied as stored.

### `apps["hasshin-kansatsu-labo"]`

Required: Yes

Nullable: No

```json
{
  "schemaVersion": 1,
  "data": {
    "entries": []
  }
}
```

Selection rule:

- Include entries whose `date` is within the last 30 days.
- Also include entries outside that range when `nextAction` is a non-empty string.

No privacy switch currently nulls hasshin-kansatsu-labo fields.

### `apps["substack-labo"]`

Required: Yes

Nullable: Yes

Current workspace shape:

```json
{
  "schemaVersion": 1,
  "data": {
    "writings": {
      "notes": [],
      "articles": []
    },
    "people": {
      "follows": [],
      "followers": []
    },
    "articleReviews": [],
    "ideas": [],
    "quickMemos": []
  }
}
```

Legacy fallback shape:

```json
{
  "schemaVersion": 1,
  "data": {
    "content": {
      "notes": [],
      "articles": [],
      "posts": []
    },
    "people": {
      "following": [],
      "followers": []
    },
    "ideas": {
      "ideas": [],
      "quick": []
    }
  }
}
```

Null condition:

- `apps["substack-labo"]` is `null` when neither `substack-labo-workspace-v2` nor the legacy fallback key contains an object value.

Privacy rule:

- `emailList` must never appear in `apps["substack-labo"]`.
- The snapshot implementation intentionally reconstructs allowed Substack sections instead of copying the whole Substack object.

### `apps["stock-labo"]`

Required: Yes

Nullable: Yes

```json
{
  "schemaVersion": 1,
  "data": {
    "items": []
  }
}
```

Null condition:

- `apps["stock-labo"]` is `null` when `privacy.stock` is `false`.

When `privacy.stock` is `true`, `items` is copied from `stock-labo-items-v1`. If the stored value is missing or not an array, `items` is an empty array.

### `apps["operation-cockpit"]`

Required: Yes for dictionaryVersion `v1.3` and later; optional for older snapshotVersion 1 dictionaries

Nullable: No

```json
{
  "schemaVersion": 1,
  "data": {
    "recentDays": {
      "2026-07-03": {
        "topPriority": "",
        "articleNote": "",
        "todayFocus": "",
        "growthTarget": "",
        "noticed": "",
        "communityChecks": {},
        "updatedAt": ""
      }
    }
  }
}
```

`recentDays` contains only valid `YYYY-MM-DD` records from the snapshot's seven-day detail period. The adapter reads `operation-cockpit-v1` without writing, repairing, or migrating it.

Only the five Intent text fields, boolean `communityChecks`, and `updatedAt` are included. Daily checks, mail checks, creator checks, and creator master data are excluded. Missing, malformed, array-based, or unrecognized legacy values produce an empty `recentDays` object.

## Required vs Optional

For snapshotVersion 1, all top-level keys listed in this document are required.

Inside `apps`, the original six app keys remain required for every snapshotVersion 1 dictionary:

- `operation-dashboard`
- `discovery-labo`
- `koryu-log-labo`
- `hasshin-kansatsu-labo`
- `substack-labo`
- `stock-labo`

`operation-cockpit` is required in snapshots generated with dictionaryVersion `v1.3` and later. Readers and validators must continue to accept older snapshotVersion 1 documents that omit it.

An app key can have a `null` value only when this document explicitly allows it.

Records inside app `data` sections preserve their source app shapes. Individual source-record fields can be optional because older localStorage records may not contain newer fields.

## Null Semantics

`null` means "intentionally unavailable" or "not computable". It must not be interpreted as an empty object, empty array, or zero.

Current null cases:

- `apps["stock-labo"]` when `privacy.stock` is `false`.
- `apps["substack-labo"]` when no readable Substack object exists.
- `operation-dashboard.data.recentDays[date].reflection` when `privacy.reflection` is `false`.
- Selected mail/DM numeric fields when `privacy.mailDmCounts` is `false`.
- Selected koryu feeling fields when `privacy.feelings` is `false`.
- `summary.longestFermentingDays` when no fermenting discovery has a valid `statusChangedAt`.

## Fields That Must Not Appear

The string key `emailList` must not appear anywhere in the generated snapshot.

This is stricter than omitting email addresses: the entire `emailList` section is excluded by structure.

## Future Extension Points

The following additions are expected to be safe when done additively:

- New top-level metadata keys, such as `generator`, `sourceUrl`, or `generatedBy`.
- New keys under `summary`, such as additional computed counts.
- New keys under `privacy`, provided old readers can ignore them.
- New app sections under `apps`, provided old readers can ignore unknown app keys.
- New fields inside existing app `data` objects, provided existing fields keep their meaning.
- A new `snapshotVersion` when the envelope shape changes in a non-additive way.
- Per-app `schemaVersion` increments when an app payload shape changes significantly.

## Backward Compatibility Rules

To preserve compatibility with existing AI prompts and future agents:

1. Do not change the meaning of existing keys.
2. Do not remove required top-level keys from snapshotVersion 1.
3. Do not rename app keys in `apps`.
4. Do not change `null` semantics. A `null` app section means intentionally unavailable, not empty.
5. Do not include `emailList` in any future snapshot version unless there is a separate explicit privacy and consent design.
6. Prefer adding new fields over changing existing fields.
7. Readers must ignore unknown keys.
8. Readers must not assume every source record has every field, because localStorage can contain older records.
9. When adding a privacy switch, document exactly which fields become `null` or which app section becomes `null`.
10. When changing a payload shape in a way that old readers cannot safely ignore, increment either `snapshotVersion` or the relevant app `schemaVersion`.

## Reader Guidance

AI agents and tools should:

- Verify `format === "sakura-snapshot"` before processing.
- Treat `mode` as a presentation hint, not as a schema selector.
- Use `privacy` to explain missing sections instead of guessing.
- Use `summary` for orientation, then consult `apps` for source detail.
- Avoid making claims from fields that are `null`.
- Avoid inferring precise elapsed time from records without a valid timestamp.
- Never require `emailList`; it is intentionally absent.

## Status

This specification is active for Phase1 and describes the current production snapshot format generated by operation-dashboard.

The format is stable for read-only AI handoff use. It is not a write-back, import, migration, or backup contract.

## AI Contract

AI agents reading this snapshot must follow these rules:

1. Treat the snapshot as read-only context.
2. Do not write changes back to any Labo or localStorage from this JSON.
3. Do not infer values from `null` sections or fields.
4. Do not request or depend on `emailList`.
5. Prefer explicit source data in `apps` over guesses from `summary`.
6. If required data is missing, ask a question instead of fabricating an answer.
7. Preserve backward compatibility by ignoring unknown future keys.
